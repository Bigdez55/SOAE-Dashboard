Honest headline: This is a real, working single-page React dashboard — the "VTA Paratransit SOAE Compliance Dashboard" — not a scaffold. ~37 first-party code files (~5,218 LOC of app code) form a fully wired chain: main.jsx → App.jsx → hooks (useDashboardData, useFilters) → dataService.js (CSV/XLSX/EML parsing + LD-penalty math) → 10 tab views rendering Recharts. It boots empty by design ("No data loaded") until you click "Load Reports"; data lives entirely client-side. The vast bulk of the tree is vendored agent tooling — atlas/ (28 MB / 3,308 files), .claude/ & .codex/ (5.6 MB each) — which has nothing to do with the app and dwarfs it. Real defects found (reported, not fixed): a double-escaped regex breaks EML date/subject extraction; the lint script can't run (eslint isn't installed); SharePoint/API data paths are dormant TODO stubs; src/data/sampleData.json is orphaned; there are no tests.

4 · Findings (reported, not fixed)

  • F-1 · EML date/subject regex is double-escaped (real bug). In dataService.js, line 730 text.match(/Subject:\\s*(.*)/i) and line 784 subject.match(/(\\d{1,2})[\\/-](\\d{1,2})[\\/-](\\d{2,4})/) use literal double-backslashes inside JS regex literals. So \\s/\\d/[\\/-] match a literal backslash char, not whitespace/digit/separator. Effect: the Subject line never matches → extractDateFromSubject receives empty input → every downed-vehicle EML record is created with date: null (and thus no yearMonth, dropping out of month filters). Both lines are defective; 784 is additionally masked because 730 already empties the subject. Confirmed via verbatim Read + grep. Fix would be single backslashes — flagged for the coding agent, not changed here.
  • F-2 · SharePoint & REST-API data paths are dormant placeholder stubs (PRESENT-BUT-NOT-WIRED). DATA_CONFIG.mode is hardcoded 'csv', so getDashboardData() always hits the default branch and returns empty data; fetchFromApi/fetchFromSharePoint are never reached, and transformSharePointData, calculateMetrics, groupDataBySource, calculateTrends are TODO placeholders returning {}/empty. README frames these as future ("for now / can be added later"), so this is possibly intentional pre-implementation — confirm intent. It means the app is strictly upload-driven.
  • F-3 · lint script is non-functional + zero tests. package.json defines "lint": "eslint . …" but eslint is absent from devDependencies and there is no .eslintrc*/eslint.config.*; npm run lint fails at command-not-found. No *.test.*/*.spec.* exist (the only "test" hits are vendored agent .md files under .claude/.codex). No CI quality gate beyond the Pages build.
  • F-4 · src/data/sampleData.json is orphaned (dead fixture). 2.3 KB file, grep shows zero imports/references anywhere in src/. Leftover sample/seed data; the app boots from getEmptyData(), not this file. Harmless but misleading.
  • F-5 · App boots to an empty dashboard with no demo data. On mount getDashboardData() returns empty and dataSource shows "No data loaded". This is by design (CSV-first, privacy-preserving), but the consequence is the deployed GitHub Pages site renders an empty dashboard for any visitor until they upload files — worth confirming that's the intended public behavior (vs. shipping a seeded demo, e.g. wiring the orphaned sampleData.json).
  • F-6 · Dead formatter exports. utils/formatters.js exports truncateText and formatDateShort; grep shows neither is imported anywhere. Minor dead code.
  • F-7 · Vendored bulk dominates the tree (honesty note, not a defect). atlas/ (28 MB / 3,308 files) plus .claude/ & .codex/ (5.6 MB each) account for ~99% of the file count against ~5,218 LOC of real app code. These are imported agent-tooling bundles, unrelated to the dashboard; raw file counts of this repo will badly overstate its size. (atlas/ is committed, not gitignored.)
  • F-8 · build:portable is identical to build. Both scripts are just vite build; "portable" in the README refers to the separate Tauri path (tauri:build). The extra script name implies a distinct portable web build that doesn't exist — low impact, naming clarity only.
