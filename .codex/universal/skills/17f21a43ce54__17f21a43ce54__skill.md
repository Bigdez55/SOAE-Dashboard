# Feature Planning Protocol

Produce a structured implementation plan before writing any code.

---

## Step 1 — Feature Brief

Ask (or infer from context):
1. **What is the feature?** One-sentence description
2. **Who uses it?** (trader, admin, automated bot, external API)
3. **What's the success metric?** (latency, accuracy, revenue, UX)
4. **Any deadline or sprint?**

## Step 2 — Impact Analysis

```bash
# Find files likely touched by this feature
# Replace KEYWORD with the feature domain (e.g., "options", "portfolio", "auth")
grep -rn "KEYWORD" backend/app/ --include="*.py" -l 2>/dev/null | head -15
grep -rn "KEYWORD" frontend/src/ --include="*.ts" --include="*.tsx" -l 2>/dev/null | head -10
```

Identify:
- **Backend:** Models, services, endpoints, schemas
- **Frontend:** Components, API hooks, store slices
- **Database:** New tables or columns (triggers `/migrate`)
- **Go microservices:** market-gateway, order-router, or risk-engine changes
- **AI/ML:** EFT agent config updates, new prompts

## Step 3 — API Contract Definition

Define the API surface before implementation:

```
ENDPOINT DESIGN
═══════════════
POST /api/v1/[resource]
Request:
  {
    "field1": type,
    "field2": type
  }
Response:
  {
    "result": type,
    "status": "success | error"
  }
Errors: 400 (validation), 401 (auth), 422 (business rule), 500
Auth: Bearer token required
```

## Step 4 — Database Schema (if needed)

```
NEW TABLE: [table_name]
═════════════════════
id           UUID PRIMARY KEY
user_id      FK → users.id
[field]      Numeric(18,8)  -- money: always Numeric, never Float
[field]      VARCHAR(N)
[field]      TIMESTAMP WITH TIME ZONE
created_at   TIMESTAMP DEFAULT NOW()
```

Flag if new columns need ALTER TABLE → route through `/migrate`.

## Step 5 — Phased Implementation Plan

```
FEATURE: [name]
═══════════════════════════════════════
PHASE 0 — Schema & Contracts (Day 1)
  □ Run ALTER TABLE if new columns
  □ Define Pydantic schemas
  □ Write API contract test (stub)
  Agent: fintech-integrity-auditor (review) + data-infra-engineer (schema)

PHASE 1 — Backend Core (Day 1-3)
  □ SQLAlchemy model update
  □ Service layer logic
  □ API endpoint + router registration
  □ Unit tests
  Agent: apex-quant-architect (trading) OR the-architect (general)

PHASE 2 — Frontend (Day 3-5)
  □ RTK Query API hook (in src/services/)
  □ React component
  □ Store integration if needed
  □ TypeScript: tsc --noEmit must pass
  Agent: product-experience-engineer

PHASE 3 — Integration & Hardening (Day 5-7)
  □ E2E integration test
  □ Error handling (400/422/500)
  □ Loading/empty/error UI states
  □ Security review if auth or money
  Agent: fintech-integrity-auditor + reliability-security-sentinel

PHASE 4 — Deploy
  □ /pre-deploy check
  □ /deploy (Iron Gate)
  □ /health verify
```

## Step 6 — Risk Assessment

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Schema drift | High if new columns | Run ALTER TABLE first |
| Breaking existing tests | Medium | Run /test after each phase |
| PII to LLM | Low | Check eft_enhance.py sanitization |
| Float for money | Low | Enforce Numeric(18,8) |
| Performance regression | Medium | Profile if >10 DB queries |
