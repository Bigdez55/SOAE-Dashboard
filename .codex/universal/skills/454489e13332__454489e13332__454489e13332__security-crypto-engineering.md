# Security & Cryptographic Engineering — 37 Skills

Implement production crypto and security hardening without third-party libraries.

## Cryptographic Implementation (11 skills)

### SK-058: SHA-256 Streaming Hash (FIPS 180-4)
Init/update/final pattern. 64-byte block processing. K[64] round constants. No libc. Self-test with NIST CAVP vectors at startup.
- **File:** `sec/xsec/crypto/sha256.c`

### SK-059: AES-256 Constant-Time S-Box
`volatile` S-box prevents compiler optimization of table lookups. Branchless, constant-time for side-channel resistance. Software fallback when AES-NI unavailable.
- **File:** `sec/xsec/crypto/aes_gcm.c:33`

### SK-060: AES-NI CPUID Detection + Hardware Dispatch
CPUID leaf 1, ECX bit 25. Runtime dispatch: AES-NI path or software path. KAT self-test at startup validates both paths.
- **File:** `sec/xsec/tls/aes_ni.c`

### SK-061: TLS 1.3 Double-Record Inbuf Staging
`ctx->inbuf[33285]`, `TLS_INBUF_CAP=2*(16384+256)+5`. Handles back-to-back TLS records without data loss.
- **File:** `sec/xsec/tls/tls13.c`

### SK-062: X25519 Montgomery Ladder with Low-Order Rejection
12 known low-order points rejected. Non-canonical input clamping. All-zero output check. Constant-time field arithmetic.
- **File:** `sec/xsec/crypto/x25519.c:480-521`

### SK-063: ECDSA P-256 Full Field Arithmetic
Scalar multiplication, signature verification. Used for kernel signing gate and TLS client auth.
- **File:** `sec/xsec/crypto/ecdsa_p256.c`

### SK-064: ChaCha20-Poly1305 (RFC 8439)
Constant-time by construction (no table lookups). 800 MB/s vs 200 KB/s for software AES-GCM. KAT verified.
- **File:** `sec/xsec/crypto/chacha20.c`

### SK-065: RSA-PSS 2048 Verify (RFC 8017)
Step-by-step PSS verification. MGF1 with SHA-256. Salt length validation. Used for TLS CertificateVerify.
- **File:** `sec/xsec/crypto/rsa.c`

### SK-066: HMAC-SHA256 + HKDF Extract/Expand
`xsec_hmac_sha256()` for MAC. HKDF-Extract (PRK from salt+IKM), HKDF-Expand (OKM from PRK+info+L). Used throughout TLS 1.3 key schedule.
- **File:** `sec/xsec/crypto/hmac.c`

### SK-067: Constant-Time Byte Comparison
`xsec_secure_compare(a, b, n)` — accumulates XOR without early exit. Prevents timing oracle on secret comparison.
- **File:** `sec/xsec/include/xsec.h`

### SK-068: xsec_secure_zero on Every Exit Path
All key material zeroed before function return. Prevents key leakage via stack inspection.
- **File:** Throughout xsec/

## Kernel Hardening (8 skills)

### SK-073: Dual-Magic Stack Canary + Guard Pages

- **File:** `kernel/xenos/core/stack_guard.c`

### SK-074: Intel VT-d IOMMU (DMAR + 4-Level IOVA)

### SK-075: XKABI HMAC-Authenticated Capability Handles
64-bit handles with bitmask rights. Bits 0-11 software, 12-22 hardware, bit 23 DERIVE. EFI handles bootstrapped at boot.

### SK-076: Safe Mode Minimal Attack Surface

### SK-077: Security Audit Ring Buffer (1024 entries)
Lock-correct ring. Overwrite tracking separated from drop counting. Covers TLS, cert, auth, camera/mic, package events.

### SK-078: Cipher Suite Hard Rejection
AES-256-GCM actively rejected in software-only TLS (timing oracle risk). ChaCha20-Poly1305 preferred. Documented condition for reversal.
- **File:** ADR-S14-07

### SK-079: 9-Gate Refusal Machine
All 9 gates run regardless of earlier failures. Complete failure report, not first-error-only. Gates 0-1 cannot be forced in production.
- **File:** `pkg/xpkg/gate/refusal_gate.c`

### SK-080: Device Permission Gate (xdev_perm_check)
Uninitialised table → EPERM (fail-safe). PID 0 always EPERM. Single-bit flag validation. boot_stubs weak version returns ALLOW — must be overridden.
- **File:** `kernel/xenos/drv/device_perm.c`, `boot_stubs.c §7`

## Branchless Patterns (3 skills)

### SK-081: Branchless Mask
`mask = -(uint32_t)(condition != 0)` — all-ones if true, all-zeros if false. Used in constant-time crypto.

### SK-082: Write-Then-Advance Ring Buffer
Write event data first, then atomically advance head pointer. Ensures consumer never sees partial data.

### SK-083: All-Zero Sentinel Detection
Check if public key is all-zeros before crypto operation. Zero-key = reject immediately (no crypto waste).
