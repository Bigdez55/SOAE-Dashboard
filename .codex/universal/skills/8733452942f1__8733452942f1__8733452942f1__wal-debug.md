# GEN.OS WAL Debug — XSTORE Crash Recovery Diagnosis

Diagnose and recover from XSTORE write-ahead log (WAL) failures in GEN.OS.

## When to use

- `xstore_open()` returns `XSTORE_ERR_CORRUPT` on boot after a crash
- B+ tree key lookups return stale or wrong values after recovery
- WAL replay counter shows fewer operations than expected
- `mvcc_snapshot_begin()` blocks or returns `XSTORE_ERR_STATE`

## XSTORE WAL Architecture

**Files:**
- `store/xstore/wal/wal.c` — WAL ring (274 MB max, 64K pages × 4096B)
- `store/xstore/txn/mvcc.c` — MVCC snapshot isolation (btree_lookup fallback)
- `store/xstore/btree/btree.c` — B+ tree (4096B nodes, 64-slot page cache)
- `store/xstore/include/xstore.h` — types: `wal_entry_t`, `xstore_txn_t`

**WAL entry format (`wal_entry_t`):**
```c
typedef struct {
    uint64_t lsn;           /* log sequence number (monotonic) */
    uint32_t txn_id;        /* owning transaction */
    uint32_t page_id;       /* target page */
    uint8_t  type;          /* WAL_INSERT / WAL_UPDATE / WAL_DELETE / WAL_COMMIT */
    uint8_t  data[4079];    /* before-image or after-image payload */
} wal_entry_t;  /* 4096B exactly — one WAL page */
```

**ARIES-style recovery (3 phases):**
1. **Analysis** — scan from last checkpoint LSN to end; rebuild dirty page table
2. **Redo** — replay all log records in LSN order (including uncommitted)
3. **Undo** — roll back all transactions with no COMMIT record (using before-images)

Reference: Mohan et al., "ARIES: A Transaction Recovery Method" (1992), ACM TODS 17(1).

## Diagnostic Steps

### Step 1 — Check WAL header integrity
```c
/* wal.h exposes: */
xstore_status_t wal_get_header(wal_ctx_t *ctx, wal_header_t *out);
/* Check: out->magic == WAL_MAGIC (0x57414C47 = "WALG") */
/*        out->version == 1 */
/*        out->lsn_head <= out->lsn_tail */
```

### Step 2 — Verify CRC32 on last 16 entries
```c
/* Each wal_entry_t has a trailing crc32 over bytes [0..4091] */
/* If CRC mismatches → partial write; truncate at last valid LSN */
uint32_t crc = pal_crc32(entry, offsetof(wal_entry_t, crc32));
if (crc != entry->crc32) { /* truncation point found */ }
```

### Step 3 — Check MVCC INSERT recovery path
The Sprint 4 fix (`store/xstore/txn/mvcc.c`) wires `btree_lookup()` fallback
for WAL_INSERT entries during recovery. Verify:
```c
/* mvcc.c recovery path — must call btree_lookup before wal_replay */
xstore_val_t existing;
xstore_status_t s = btree_lookup(ctx->btree, entry->key, &existing);
if (s == XSTORE_OK && existing.txn_id == entry->txn_id) {
    /* already applied — skip */
}
```

### Step 4 — Run WAL replay with verbose logging
```c
/* Enable PAL_CONSOLE_VERBOSE before xstore_open() in debug build */
#define XSTORE_WAL_VERBOSE 1
xstore_status_t s = xstore_open(ctx, path, XSTORE_OPEN_RECOVER);
/* Check return: XSTORE_OK | XSTORE_ERR_CORRUPT | XSTORE_ERR_TRUNCATED */
```

## Common Failure Patterns

| Symptom | Root Cause | Fix |
|---------|-----------|-----|
| `XSTORE_ERR_CORRUPT` on open | CRC32 mismatch in last N entries | Truncate at last valid LSN, replay |
| Stale reads after recovery | btree_lookup not wired in MVCC INSERT path | Apply Sprint 4 mvcc.c fix |
| WAL ring full (274 MB) | Checkpoint not triggered | Call `wal_checkpoint(ctx)` to flush + reclaim |
| `mvcc_snapshot_begin()` hangs | write_lock held by crashed txn | Force-release: `pal_spin_unlock(&ctx->write_lock)` |
| Wrong value after rollback | Before-image not stored for UPDATE | Verify `wal_entry_t.type == WAL_UPDATE` uses before-image in `data[]` |

## Recovery Command Sequence

```c
/* 1. Open in recovery mode */
xstore_ctx_t ctx;
xstore_open(&ctx, "/xstore/main.db", XSTORE_OPEN_RECOVER);

/* 2. Query WAL stats */
wal_stats_t stats;
wal_get_stats(&ctx.wal, &stats);
/* stats.replayed_entries, stats.aborted_txns, stats.last_valid_lsn */

/* 3. Verify tree integrity post-recovery */
uint64_t corrupt_nodes = btree_verify(&ctx.btree);
/* Must be 0 before accepting the recovered store */

/* 4. Checkpoint to compact WAL */
wal_checkpoint(&ctx);
```
